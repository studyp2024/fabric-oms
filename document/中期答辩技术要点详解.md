## 1. 系统的整体运行流程（数据流转）

在向老师介绍时，可以用一条“**运维人员敲下一行命令后，系统做了什么？**”的线索来串联整个架构。

### 第一步：日志的产生（目标服务器）
当运维人员通过 SSH 登录到目标服务器（如 192.168.1.100）并执行一条命令（例如 `rm -rf /tmp/test`）时，目标服务器上预先配置好的 `PROMPT_COMMAND` 脚本会被触发。
这个脚本会实时抓取刚才执行的命令，连同**时间戳**、**客户端IP**、**当前用户**、**当前目录**等信息，拼接成一条特定格式的字符串，并追加写入到 `/var/log/ssh_commands.log` 文件中。

### 第二步：日志的采集（后端定时任务）
我们的 OMS 系统后端（Spring Boot）中有一个定时任务。它每隔 10 秒钟就会醒来一次，执行以下操作：
1. 从 MySQL 数据库中查出所有纳管的服务器列表。
2. 拿到这台服务器的 IP 和 SSH 密码，使用 `JSch` 库在后台**建立一个 SSH 通道**连接过去。
3. 它不会把整个日志文件下载下来（太浪费带宽），而是从数据库里查出上次读到了多少个字节（`offset`），然后向服务器发送类似 `tail -c +[offset+1] /var/log/ssh_commands.log` 的命令，**只拉取增量的新日志**。

### 第三步：数据的双写（MySQL + 区块链）
后端拿到这条新增的日志后，会进行解析和处理：
1. **计算哈希**：把日志内容和服务器IP拼在一起，用 SHA-256 算法计算出一个唯一的 Hash 值。这个 Hash 值就像是这条日志的“指纹”。
2. **上链存证**：后端通过 `Fabric Gateway SDK`，把这条日志的数据和它的 Hash 值打包，作为一笔交易提交给 Hyperledger Fabric 联盟链。链码（智能合约）会判断这条命令是不是高危操作，然后把它永久写入区块账本中，不可篡改。
3. **关系型存储**：上链成功后，后端把这条记录也存入 MySQL 的 `audit_logs` 表中。这样做是为了前端 Vue 页面能快速地进行分页查询和全文检索（因为直接从区块链查大量数据很慢）。
4. **更新指针**：最后，把刚才读取的字节数加到 `offset` 上存回 MySQL，保证下次采集不重不漏。

## 2. 核心功能的技术实现细节

### 2.1 增量日志采集怎么做的？
**痛点**：如果每次都全量读取日志，文件大了会导致内存溢出和网络拥堵。
**解决**：在 `LogSyncService.java` 中。
```java
// 核心代码：LogSyncService.java
long offset = server.getLastLogOffset();
// 动态拼接 tail 命令，只读取 offset 之后的数据
String command = "tail -c +" + (offset + 1) + " " + logFilePath;
channel.setCommand(command);

// ... 读取流 ...
while ((line = reader.readLine()) != null) {
    // 计算本次读取了多少字节
    byte[] lineBytes = (line + "\n").getBytes(StandardCharsets.UTF_8);
    bytesRead += lineBytes.length;
    // 处理日志
    processLogLine(line, server.getIp());
}
// 更新数据库里的 offset 指针
server.setLastLogOffset(offset + bytesRead);
serverInfoRepository.save(server);
```
**向老师解释**：我们采用了记录偏移量（Offset）结合 Linux 的 `tail -c` 命令的方法。MySQL 记录了每台服务器读到了第几个字节，每次只读新增的部分，这种轻量级的探针方式不会影响目标服务器的性能。

### 2.2 区块链防篡改是怎么落地的？
**痛点**：传统的运维审计系统日志存在本地数据库，拥有数据库 root 权限的人可以轻易删改日志（比如删库跑路后抹除痕迹）。
**解决**：Fabric 联盟链 + 双写机制。
在 `FabricService.java` 中，我们使用了 Fabric 官方提供的 Gateway SDK：
```java
// 核心代码：FabricService.java
byte[] result = contract.submitTransaction("CreateLog",
        log.getId(),        // 唯一ID(基于Hash)
        log.getTimestamp(),
        log.getIp(),
        log.getServerIp(),
        log.getUser(),
        log.getPwd(),
        log.getCommand(),
        log.getHash());     // 核心防篡改依据：原始数据的 Hash
```
在链码 `audit.go` 中，接收到数据后存入世界状态（World State）：
```go
// 核心代码：audit.go (链码)
logJSON, err := json.Marshal(log)
// PutState 将数据写入 Fabric 账本，一旦写入，会在所有 Peer 节点共识同步，无法被单一节点篡改
err = ctx.GetStub().PutState(id, logJSON)
```
**向老师解释**：我们将数据的哈希值和关键字段写入 Fabric 账本。由于区块链的分布式共识和密码学特性，这些数据一旦上链就无法被修改或删除。即使黑客攻破了 MySQL 修改了展示数据，系统在对账时，只要将 MySQL 的数据重新计算 Hash 与链上比对，就能立刻发现篡改行为。

### 2.3 高危操作（敏感命令）是如何预警的？
我们在区块链的**智能合约（链码）**层面直接写死了高危操作的判断逻辑，而不是写在后端。
```go
// 核心代码：audit.go (链码)
func checkSensitive(command string) bool {
	// 定义敏感命令的正则表达式
	sensitivePatterns := []string{
		`\brm\b`,      // 删除文件
		`\bsudo\b`,    // 提权
		`\bchmod\b`,   // 改权限
		`\bpasswd\b`,  // 改密码
		// ... 其他危险命令
	}
	for _, pattern := range sensitivePatterns {
		matched, _ := regexp.MatchString(pattern, command)
		if matched {
			return true // 命中敏感词，返回 true
		}
	}
	return false
}
```
在 `CreateLog` 链码函数被调用时，会自动调用 `checkSensitive`，把 `isSensitive` 字段标记为 `true` 存入账本。
**向老师解释**：把敏感词判断写在智能合约（链码）里的好处是**规则透明且不可篡改**。如果有人想偷偷把 `rm` 命令移出敏感词库，他必须发起链码升级提案，这需要联盟链中各个组织的背书同意，杜绝了单方面掩盖高危操作的可能。前端拿到 `isSensitive=true` 的数据后，会用红色高亮标出，起到预警作用。

## 3. 常见答辩问题预演 (Q&A)

**Q1：你的系统为什么既用 MySQL 又用 区块链（Fabric）？只用区块链不行吗？**
> **答**：区块链的强项是防篡改和可信存证，但它的弱点是查询性能低，尤其是做类似“模糊搜索某段时间内某个IP的特定命令”这种复杂查询时，Fabric 的 CouchDB 性能远不如关系型数据库。
> 因此我们采用“双写”架构：MySQL 作为链下快照，负责支撑前端高并发、复杂条件的分页展示；Fabric 作为信任根，负责存储数据的 Hash 和核心快照。展示靠 MySQL，信任靠 Fabric。

**Q2：如果服务器断网了，或者日志同步服务挂了，日志会丢吗？**
> **答**：不会。目标服务器产生的日志是先实实在在地追加写入本地的 `/var/log/ssh_commands.log` 文件中的。我们的采集服务是基于 `offset`（偏移量）的。如果断网，偏移量不会更新；等网络恢复或服务重启后，定时任务会继续从上次记录的 `offset` 位置开始读取，补齐中间落下的所有日志，保证数据不丢失。

**Q3：你这个系统在安全性上除了区块链还有什么设计？**
> **答**：我们在系统内做了严格的权限隔离。通过 JWT (JSON Web Token) 鉴权，系统分为 Admin（管理员）和 Ops（运维）角色。Ops 登录后，只能看到被分配给自己的服务器资产，并且在查日志时，后台会在 SQL 层面强制加上 `assigned_user_id` 的过滤，防止越权越级查看其他服务器的审计记录。同时，系统的 Fabric 交互依赖严格的 X.509 证书体系，确保上链身份的合法性。
